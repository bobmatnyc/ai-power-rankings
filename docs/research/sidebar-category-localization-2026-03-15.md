# Sidebar Category Localization Investigation

**Date:** 2026-03-15
**Status:** Complete
**Finding:** Categories are NOT localized — raw English strings from DB/static file are rendered directly

---

## 1. The Sidebar Component

**File:** `/Users/masa/Projects/aipowerranking/components/layout/app-sidebar.tsx`

The left-rail sidebar is rendered by `AppSidebar` -> `SidebarContent`.

The component DOES use `useI18n()` (line 38) and correctly localizes:
- Navigation labels (lines 47–75) via `dict.navigation.*`
- Section headings like "Categories", "Features", "Navigation" (lines 158, 216, 253) via `dict.sidebar.*`
- Tag/feature filter labels (lines 127–133) via `dict.features.*`

**But category names themselves are NOT localized.** Line 296 renders the raw `category.name` string:

```tsx
// app-sidebar.tsx, line 296
<span className="transition-colors">{category.name}</span>
```

No `dict` lookup is performed on `category.name`. It renders whatever string is stored in the `Category` object.

## 2. Where Category Names Come From

### Data flow:
```
lib/data/static-categories.ts (STATIC_CATEGORIES)
  -> app/[lang]/layout.tsx (line 116, passed as categories prop)
  -> ClientLayout (client-layout.tsx, line 71)
  -> AppSidebar (app-sidebar.tsx, line 379)
  -> rendered at line 296 as {category.name}
```

### Static categories file
**File:** `/Users/masa/Projects/aipowerranking/lib/data/static-categories.ts`

This file is auto-generated at build time and currently contains only:
```ts
export const STATIC_CATEGORIES: Category[] = [
  { "id": "all", "name": "All Categories", "count": 0 }
];
```

The `name` field is hardcoded English. It is generated by the `getCategoryName()` function in the categories repository (see below).

### Categories repository
**File:** `/Users/masa/Projects/aipowerranking/lib/db/repositories/categories.ts`

`getCategoryName()` (lines 25–30) converts raw category IDs to English display names by capitalizing words:

```ts
function getCategoryName(categoryId: string): string {
  return categoryId
    .split("-")
    .map((word) => word.charAt(0).toUpperCase() + word.slice(1))
    .join(" ");
}
```

The "All Categories" fallback (lines 41, 53, 95, 109) is also hardcoded English.

## 4. Diagnosis: What Is Broken and Why

### Root Cause
The `Category` objects passed into the sidebar carry a pre-computed English `name` string (set at build time or DB query time). The sidebar renders `{category.name}` directly without consulting the i18n dictionary.

There are TWO places where the English name is baked in:

1. **`lib/db/repositories/categories.ts` line 86** — `name: getCategoryName(id)` — converts DB category IDs to English names at query time.
2. **`lib/data/static-categories.ts` line 20** — `"name": "All Categories"` — English hardcoded in the generated static file.

The `dict.categories.*` i18n keys exist and contain translations, but no code path uses them for sidebar rendering.

### Secondary Issue
The `static-categories.ts` file currently only contains the "All Categories" placeholder (count: 0), suggesting the generate script hasn't run successfully against the live DB. When it does run, all generated names will be English-only strings from `getCategoryName()`.

---

## 5. What the Fix Should Be

The fix is in `app-sidebar.tsx`. Instead of rendering `{category.name}`, look up the localized name from `dict.categories` using `category.id` as the key, falling back to `category.name` if no translation exists.

**Current code (line 296):**
```tsx
<span className="transition-colors">{category.name}</span>
```

**What it should do:**
```tsx
<span className="transition-colors">
  {category.id === "all"
    ? (dict.navigation.allCategories ?? category.name)
    : (dict.categories?.[category.id] ?? category.name)}
</span>
```

Additionally, the i18n keys in `dict.categories` need to match the actual category IDs stored in the DB. The en.json has both camelCase (`autonomousAgent`) and lowercase (`autonomousagent`) variants — whichever matches the real DB IDs should be kept; the other can be removed.
